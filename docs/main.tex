\begin{center}
    {\Huge \textbf{ABSTRACT}}\par
    \vspace{1.5cm}
\end{center}

\begin{spacing}{1.5}
Inter-district transportation in Tier-3 Indian regions, specifically Barak Valley, Assam, relies heavily on analogue, highly fragmented coordination methods. Fleet operators manage bookings via physical ledgers and phone calls, leading to pervasive double bookings and unutilised capacity. Customers, conversely, lack any digital discovery mechanism to query available routes or secure seat reservations, resulting in significant friction. \textit{Pather Saathi} resolves this gap by providing an ultra-localised, cloud-native fleet booking platform tailored to the constraints of regional demographics. 

The platform implements atomic booking allocation using robust PostgreSQL Row Level Security (RLS) policies and Role-Based Access Control (RBAC), completely eliminating concurrency issues. Unlike metropolitan solutions that force proprietary app adoption, Pather Saathi adopts a WhatsApp-first interaction model, leveraging deep-linking to route booking confirmations and operator communications through the universally adopted messenger layer.

Methodologically, this project validates a multi-model AI-augmented development pipeline. Using the Model Context Protocol (MCP), AI agents were orchestrated to accelerate implementation while the human engineering team focused on architectural design and rigorous adversarial security audits. The final product was successfully deployed using a serverless architecture (Vercel and Supabase), achieving a highly scalable production environment with minimal monthly infrastructural cost. The result is a cost-effective web platform suitable for digital fleet management in underserved regions.

\vspace{1cm}
\textbf{Keywords:} Fleet Booking, Barak Valley, Serverless Architecture, Row Level Security, AI-Augmented Development, Model Context Protocol, WhatsApp Integration.
\end{spacing}

\chapter{Introduction and Objectives}

\section{Background}
The digitisation of transportation infrastructure is an essential component of economic mobility in developing regions. In Tier 3 Indian geographies, particularly in areas characterised by complex topographies and underdeveloped digital ecosystems, the gap between available technology and local adoption remains significant. While metropolitan areas benefit from integrated mobility-as-a-service (MaaS) applications, rural and semi-urban districts rely on informal methods for transit coordination. The infrastructural context in Northeast India, particularly Barak Valley, is challenging. Despite high smartphone penetration, the local transport sector remains technologically isolated, presenting an opportunity for formal digitisation.

\section{About the Project}
``Pather Saathi'' is a web-based fleet booking platform developed for the socioeconomic and technological realities of Barak Valley. The platform serves two primary user cohorts: local fleet operators and the passengers. Geographically, the project targets the transit corridors connecting Silchar, Hailakandi, and Karimganj (Sribhumi). By leveraging communication channels like WhatsApp for confirmations alongside a Progressive Web Application (PWA) interface, the platform digitises the booking lifecycle. It enables customers to discover available routes and request seats, while providing operators with a cloud-based dashboard to manage their fleet and approve bookings.

\section{Objectives}
The primary objectives of this project are:
\begin{itemize}[leftmargin=*]
    \item To design and implement a web-based fleet booking platform tailored for Barak Valley operators and customers.
    \item To enable real-time seat availability tracking to resolve the systemic issue of double bookings.
    \item To implement a secure, multi-tenant operator dashboard enforcing strict Role-Based Access Control (RBAC).
    \item To evaluate an AI-augmented development methodology leveraging Model Context Protocol (MCP) tooling.
    \item To deploy a reliable system using a serverless architecture with minimal infrastructure overhead.
    \item To validate the system's security posture via rigorous testing of database policies.
\end{itemize}

\section{Key Contributions}
This project provides several key technical contributions to the field of localised software engineering:
\begin{itemize}[leftmargin=*]
    \item \textbf{Serverless Architecture:} Designed a scalable fleet management system using Next.js and Supabase.
    \item \textbf{WhatsApp Deep-Link Protocol:} Implemented a seamless booking confirmation flow utilizing \texttt{wa.me} dynamic URI generation.
    \item \textbf{Atomic Seat Allocation:} Implemented PostgreSQL transactions and Row Level Security to strictly enforce tenant isolation and prevent double bookings.
    \item \textbf{AI-Augmented Methodology:} Evaluated a multi-model development pipeline utilizing MCP to assist in system implementation.
    \item \textbf{Tier 3 UX Optimizations:} Developed a lightweight SSR frontend tailored specifically for mid-range Android devices on variable 4G connections.
\end{itemize}

\chapter{Problem Statement}

\section{Current Situation}
The current state of inter-district transport coordination in Barak Valley relies entirely on analogue communication systems. Fleet bookings are executed predominantly through telephone calls and physical ledger entries. Operators manage their daily inventory using notebooks at physical transit hubs. Passengers rely on personal connections or physical visits to ascertain vehicle availability, departure times, and pricing. 

\section{Operator Problems}
For fleet operators, these manual processes manifest as severe operational pain points. The most critical issue is the high frequency of double bookings due to human error in ledger management. Operators also suffer from idle fleet capacity without a centralised digital broadcasting mechanism. The lack of a unified dashboard limits real-time visibility into projected revenue or fleet utilisation metrics.

\section{Customer Problems}
Conversely, customers face significant friction when interacting with this analogue system. The primary pain point is the complete absence of a digital discovery mechanism. Once a booking is verbally agreed upon, there is no formal digital confirmation, leading to uncertainty. Customers lack any mechanism for tracking the status of their booking or securely managing their travel itineraries.

\section{Limitations of Existing Systems}
While generic booking platforms and large-scale aggregators exist in metropolitan markets, they fail to fit the context of Barak Valley. Enterprise solutions impose prohibitive subscription costs and require operators to possess advanced digital literacy. National aggregators extract significant commission percentages, counter to the low-margin reality of local fleet owners. Furthermore, these platforms attempt to force users into proprietary in-app messaging systems, ignoring that WhatsApp is the accepted communication standard in this demographic.

\section{Formal Problem Statement}
The inter-district road transport sector in Barak Valley suffers from critical operational inefficiencies due to a reliance on fragmented, analogue booking methods. This leads to high rates of double bookings, unutilised fleet capacity, and customer friction in discovering transit options. There is a requirement for an ultra-localised digital fleet management platform that digitises discovery and guarantees atomic seat allocation while aligning with the region's existing digital literacy and reliance on WhatsApp communication.

The digital transformation of transportation networks across India has historically been concentrated in Tier 1 and Tier 2 cities, driven by the rapid expansion of mobility-as-a-service (MaaS) platforms. \chapter{Proposed Solution}

\section{Solution Overview}
Pather Saathi is developed as a two-sided, web-based platform built on a modern, serverless Next.js architecture. The solution digitises the booking lifecycle by providing a customer-facing portal for route discovery and seat reservation, integrated seamlessly with a secure, multi-tenant dashboard for fleet operators. By centralising inventory data in a PostgreSQL database managed via Supabase, the system implements atomic transactions to mitigate the possibility of double bookings.

\section{WhatsApp Integration}
A critical element of the proposed solution is its WhatsApp-first communication strategy. Pather Saathi utilises dynamic deep-linking via the \texttt{wa.me} protocol. When a customer initiates a booking, the system generates a secure reference ID and constructs a pre-filled WhatsApp message. This leverages existing digital habits while maintaining a verifiable digital paper trail within the platform.

\chapter{Methodology and Working Process}

\section{Requirement Analysis}
The project commenced with an analysis of the operational constraints in Barak Valley. This phase identified the critical need for a low-friction booking system that required minimal digital literacy, leading to the selection of WhatsApp as the primary communication medium.

\section{UI/UX Design}
A mobile-first design strategy was adopted, acknowledging that the primary user base accesses the internet via mid-range Android devices. The interface was constructed using Shadcn UI components to ensure accessible touch targets and high-contrast visibility under direct sunlight.

\section{Database Design}
The data layer was modeled using PostgreSQL. The schema was normalized to support multi-tenancy, ensuring that fleet operators only have access to their specific vehicles and routes. Row Level Security (RLS) policies were designed as the primary defense mechanism against unauthorized data access.

\section{Implementation}
The system was developed utilizing a modern Next.js framework. To accelerate implementation, the engineering team leveraged AI-assisted code generation via the Model Context Protocol (MCP). While AI agents assisted with boilerplate and repetitive logic, the human team maintained strict oversight over architectural decisions and security implementations.

\section{Testing}
Testing encompassed strict build verification, adversarial Row Level Security (RLS) audits to verify database isolation, and manual User Acceptance Testing (UAT) on physical devices to validate WhatsApp deep-linking functionality across varying network conditions.

\section{Deployment}
The application was deployed on Vercel's serverless edge infrastructure to ensure high availability and minimal latency, with the PostgreSQL database hosted on Supabase.

\section{Security Architecture}
The security architecture operates on a defense-in-depth model. The primary defence mechanism is PostgreSQL Row Level Security (RLS). Every table possesses strict \texttt{SELECT}, \texttt{INSERT}, \texttt{UPDATE}, and \texttt{DELETE} policies. Next.js Server Actions act as the secondary trust boundary, sanitising and validating all input parameters before interacting with the database.

\chapter{Challenges Faced}

\begin{challengebox}[Infinite Recursion in Row Level Security (RLS) policies]
\textbf{Solution:} The policy evaluated recursively when joining the \texttt{bookings} table with the \texttt{users} table. This was resolved using \texttt{SECURITY DEFINER} PostgreSQL functions to safely bypass the policy evaluation loop.

\vspace{0.5em}\noindent\textbf{Lesson Learned:} Direct table joins within RLS policies must be avoided in favour of dedicated security functions.
\end{challengebox}

\begin{challengebox}[Ephemeral State in Serverless Rate Limiting]
\textbf{Solution:} In-memory rate limiting failed on Vercel due to isolated, ephemeral serverless instances. This necessitated architecting a distributed rate-limiting solution using Upstash Redis.

\vspace{0.5em}\noindent\textbf{Lesson Learned:} Global state cannot be maintained in serverless environments without an external caching layer.
\end{challengebox}

\begin{challengebox}[Performance on Low-End Devices]
\textbf{Solution:} Initial field testing exhibited sluggish performance due to large client-side JavaScript bundles. The team resolved this by aggressively refactoring the component tree to utilise React Server Components (RSC).

\vspace{0.5em}\noindent\textbf{Lesson Learned:} Mobile-first design must prioritize minimizing client-side hydration payloads for Tier-3 network constraints.
\end{challengebox}

\begin{challengebox}[AI-Generated Code Quality Control]
\textbf{Solution:} While AI improved velocity, early iterations suffered from subtle logical flaws, requiring secondary adversarial review passes.

\vspace{0.5em}\noindent\textbf{Lesson Learned:} AI agents are powerful execution engines but require human engineers to act as strict architectural validators.
\end{challengebox}

\chapter{Conclusion}

The development and deployment of the Pather Saathi platform yield the following primary conclusions:
\begin{enumerate}[leftmargin=*]
    \item \textbf{Digitisation of Informal Transit:} Hyper-localised platforms tailored to regional constraints are highly effective at formalising analogue transport networks, bridging critical infrastructural gaps in developing regions.
    \item \textbf{Efficacy of WhatsApp Integration:} Leveraging existing digital habits, such as WhatsApp for communication, significantly reduces friction and accelerates user adoption in demographics with varying digital literacy.
    \item \textbf{Serverless Multi-Tenancy:} A serverless architecture combined with strict PostgreSQL Row Level Security (RLS) provides a robust, scalable foundation for ensuring strict tenant isolation among independent fleet operators.
    \item \textbf{Modern Development Workflow:} The integration of Model Context Protocol (MCP) tooling and AI agents dramatically increases development velocity, validating the future role of human engineers as strategic orchestrators.
    \item \textbf{Mobile-First Performance:} Strict adherence to Server-Side Rendering (SSR) and minimal client-side bundles is an absolute prerequisite for applications targeting low-end Android devices on variable 4G networks.
\end{enumerate}
